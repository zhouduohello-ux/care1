# CareMemory Product Requirements Document (PRD)

> **版本**：v0.2 — AI-native 架构草案  
> **日期**：2026-06-15  
> **目标市场**：英国（UK）  
> **目标疾病**：哮喘（Asthma）  
> **IM 入口**：WhatsApp  
> **文档状态**：待技术评审

---

## 1. 产品概述

CareMemory 是一款 **AI-native 的复诊前健康记忆引擎**。患者在两次复诊之间通过日常使用的即时通讯工具（首个 MVP 为 WhatsApp）进行轻量、对话式的健康记录；引擎基于医学知识库、疾病管理策略和患者实际回答，动态决定每次该问什么、如何追问、何时提醒，最终生成一页 **Disease Card（疾病卡片）** 及其复诊导出形态 **Brief**。

CareMemory 不是日记，也不是量表工具。它的核心能力是把稀疏、对话式的患者自报信息，转化为对患者和医生都有参考价值的结构化健康叙事。

### 1.1 一句话定位

> 用 WhatsApp 轻量记录，AI 帮你整理成一张清晰、科学的疾病卡片，复诊前自动生成医生可看的摘要。

### 1.2 核心价值主张

- **对患者**：不再靠回忆描述过去几周甚至几个月的变化，能看到自己的疾病故事、依从性、主观感受变化和想问医生的问题。
- **对医生**：60 秒内获得患者自报的结构化摘要，辅助问诊，而不是替代临床判断。
- **对产品**：验证「IM-first + AI-native 决策引擎 + Disease Card + Doctor Brief」这个最小闭环是否成立。

### 1.3 产品边界（MVP 不做）

| 不做 | 原因 |
|------|------|
| 诊断或治疗建议 | 医疗合规红线，仅做记录、整理与呈现 |
| 判断是否需要急救 | 只能引导患者遵循医生已给行动计划或拨打急救电话 |
| 连接医疗器械（峰流速仪、吸入器传感器、CGM 等） | 超出 MVP 范围，未来扩展 |
| 医生端入驻或 NHS 集成 | 太重，MVP 仅通过 Web Brief / PDF 传递 |
| 多语言 / 多病种 / 多平台落地 | 首次验证只聚焦英国 + 哮喘 + WhatsApp，但架构必须可扩展 |
| 从非量表问题推断临床量表分数 | 不能对稀疏非结构化回答做临床评分；仅当问题本身为量表时才输出量表值 |

## 2. 目标用户

### 2.1 主要用户：英国成人哮喘患者

- 年龄：18–60 岁；
- 使用 WhatsApp 作为日常通讯工具；
- 每 1–3 个月需要复诊（GP / Asthma Nurse / Respiratory Specialist）；
- 复诊前常感到「说不清楚最近控制得怎么样」；
- 愿意在两次复诊之间每周花 1–2 分钟做轻量记录。

### 2.2 次要用户：医疗专业人士

- GP、哮喘护士、呼吸科医生；
- 不直接在产品中注册，通过患者分享的 Web Brief / PDF 间接接触；
- 需要看到结构清晰、不越界的自报信息。

## 3. MVP 范围

首个 MVP 聚焦一个最小组合：

> **英国 × 哮喘 × WhatsApp**

但产品架构必须支持未来多病种、多 IM 平台、多市场的扩展。MVP 是「用哮喘验证引擎」，而不是「做一个哮喘问卷工具」。

### 3.1 功能范围

| 模块 | MVP 内 | MVP 外 |
|------|--------|--------|
| 患者接入 | WhatsApp Business API 接入，用户扫码/点击开始 | LINE、SMS、iMessage 等其它 IM |
| 记录方式 | AI 驱动的聊天式轻问答（每 2 天 1 次，常规最多 3 个问题；异常时可覆盖） | 自由文本日记、语音、图片 |
| 记录内容 | 夜间症状、急救吸入药使用、活动受限、诱因、依从性、主观感受、不良反应、想问医生的问题 | 峰流速数值、血氧、设备同步 |
| 数据查看 | 安全的 Web/PWA 页面查看 Disease Card 和历史记录 | 原生 App |
| Disease Card | 早期版本：7 天摘要 + 趋势雏形 + 患者问题 | 完整的 1–3 个月跨周期分析 |
| Brief 生成 | 7 天后生成一页式 Asthma Visit Brief（Web + PDF） | 医生端 Dashboard |
| 分享方式 | 患者复制链接或下载 PDF 带到复诊 | 直接发给医生系统 |
| 付费 | 4 周计划预订按钮（用于验证付费意愿） | 真实订阅系统、退款流程 |

### 3.2 记录周期

- **7 天试用周期**：用于冷启动验证，生成早期 Disease Card 雏形和 Brief；
- **4 周正式计划**：用于付费意愿验证，开始展现更有价值的趋势；
- **真实复诊周期**：1–3 个月， Disease Card 会随时间积累而更有医学参考价值。

## 4. 用户旅程

### 4.1 首次使用（Day 0）

1. 用户通过落地页、社群或广告看到信息；
2. 点击 CTA 后跳转到 WhatsApp，发送预设开场消息（如 `START ASTHMA`）；
3. 系统回复欢迎语，说明：
   - 这不是诊断工具；
   - 数据仅用于生成 Disease Card 和复诊摘要；
   - 敏感记录存放在安全 Web 页面；
4. 用户确认参与并同意隐私条款（链接到 Web 页面）；
5. 系统收集基础信息：昵称、下次复诊日期（可选）、时区；
6. 创建第一个 Care Cycle，进入 ACTIVE 状态。

### 4.2 日常记录（Day 1–7 及以后）

1. 系统每 2 天发起一次主动 check-in；
2. AI Planner 根据患者当前状态、历史记录、疾病策略、时间上下文，决定本次要探查的疾病片段和最多 3 个问题；
3. 用户通过选项按钮或简短回复回答问题；
4. 每答一题，Planner 重新评估：继续原计划、追问异常、或提前结束；
5. 敏感或详细记录建议用户到 Web/PWA 页面查看和编辑；
6. 用户也可主动发起咨询，不占用 check-in 预算。

### 4.3 Disease Card 查看

1. 每次 check-in 后，患者记忆层和 Disease Card 同步更新；
2. 用户可在 Web/PWA 查看自己的 Disease Card；
3. 7 天试用后展示早期版本：周期摘要、关键信号、患者问题；
4. 4 周及以后展示更完整的趋势和模式洞察。

### 4.4 Brief 生成（复诊前）

1. 在周期结束或复诊前，系统从 Disease Card 截取并编排生成一页式 Asthma Visit Brief；
2. 通过 WhatsApp 发送 Web Brief 链接；
3. 用户可在 Web 页面查看、编辑备注、下载 PDF；
4. 用户带到复诊或分享给医生。

### 4.5 转化（Day 7–14）

1. 系统询问用户是否愿意继续 4 周计划；
2. 展示预订按钮和 early pilot price（£19）；
3. 收集预订意向（不一定真实扣款）。

## 5. 核心架构

CareMemory 是一个分层的 AI-native 引擎。MVP 只实现 asthma profile，但架构必须支持未来加载 IBD、糖尿病/高血压等 profile。

### 5.1 六层引擎

| 层级 | 名称 | 职责 |
|------|------|------|
| L1 | 感知层 | 解析任何用户输入，提取信号、意图、情绪、异常标记 |
| L2 | 患者记忆层 | 维护不可变事件日志、观察记录、多级 narrative summary、Disease Card |
| L3 | 策略库层（RAG Corpus） | 提供医学知识、疾病管理策略、对话示例、安全规则 |
| L4 | 规划层 | 决定当前该推进哪个疾病片段、问什么问题、是否进入异常流程 |
| L5 | 对话层 | 把规划意图渲染为 WhatsApp 消息，管理轮次与交互形态 |
| L6 | 安全/合规层 | 校验输出，拦截越界表述，追加标准安全提示 |

### 5.2 四层 RAG Corpus

- **Medical Knowledge Base**：疾病管理知识，用于理解患者输入和生成解释；
- **Care Strategy Library**：CareMemory 特有的疾病管理策略，指导 Planner 决策；
- **Conversation Pattern Library**：高质量问题、追问、回复示例，指导对话层生成；
- **Safety Rules**：安全边界和禁止表述。

> 注：PRD v0.1 中称其为“三层”，实际包含四个子库，v0.2 已修正。

### 5.3 IM 兼容层

- 引擎内部使用平台无关的消息模型；
- IM Adapter 负责与 WhatsApp / LINE 等平台对接；
- 引擎输出「语义交互意图」，Adapter 决定具体交互组件。

### 5.4 Disease Card 与 Brief 的关系

- **Disease Card** 是患者的长期疾病肖像，持续更新，患者主看；
- **Brief** 是 Disease Card 在复诊前的导出形态，医生主看；
- Brief 从 Disease Card 中截取、编排、强调 visit-relevant 信息。

## 6. 核心功能需求

### 6.1 IM 会话管理

- 支持通过 WhatsApp Business API 接收和发送消息；
- 维护用户业务会话状态（onboarding、active、brief_ready、completed）；
- 支持按时间触发的检查点消息；
- 支持按钮式、列表式、自由文本式回复；
- 处理 24 小时会话窗口：窗口内自由消息，窗口外使用预审批模板；
- 支持 STOP / HELP / DELETE MY DATA 等系统指令。

### 6.2 AI 驱动的记录采集

- 不依赖硬编码问题清单；
- Planner 基于患者状态、疾病策略、时间上下文动态决定问题；
- 支持结构化的选择题、评分题、简短文本输入；
- 每答一题后可重新规划下一步；
- 异常或安全信号可突破 3 题预算；
- 采集内容包括但不限于：
  - 症状（夜间咳嗽/喘息、活动受限等）
  - 用药/治疗依从性（急救吸入药使用、控制器使用等）
  - 诱因/暴露（花粉、灰尘、冷空气、运动、感染等）
  - 主观感受变化
  - 不良反应
  - 想问医生的问题

### 6.3 Disease Card

- 基于患者记忆层持续更新；
- 7 天试用周期展示早期版本：周期摘要、关键信号、患者问题；
- 4 周及以后展示更完整版本：依从性、症状-治疗关系、不良反应、主观趋势、模式洞察；
- 不推断临床量表分数；仅当问题本身为量表时才展示量表值；
- 患者可在 Web/PWA 查看，不可直接编辑 LLM 生成的洞察，但可修正底层记录。

### 6.4 Web Brief 生成

- 从 Disease Card 截取并编排生成；
- 一页式摘要，包含：
  - 标题与周期
  - 控制状态/关注要点
  - 关键信号趋势
  - 用药/依从性摘要
  - 不良反应/异常事件
  - 患者问题清单
  - 免责声明
- 生成 Web 页面和 PDF 两种格式；
- Web 页面需要密码/一次性链接保护。

### 6.5 用户管理

- 通过 WhatsApp 电话号码唯一识别用户；
- 支持用户查看、编辑自己的底层记录；
- 支持用户删除账户和数据（GDPR 要求）。

### 6.6 运营与反馈

- 收集用户完成率、记录质量、Disease Card 打开率、Brief 打开率；
- 收集医生/医疗专业人士对 Brief 的反馈；
- 支持 A/B 测试不同策略和对话风格；
- 记录 LLM 推理痕迹用于审计和优化。

## 7. 非功能需求

### 7.1 安全与合规

- 所有健康数据传输使用 TLS；
- 静态数据加密存储；
- 不与第三方共享健康数据；
- 明确告知用户产品不是诊断工具；
- 哮喘相关消息必须包含标准安全提示；
- 遵守 UK GDPR / GDPR 数据最小化原则；
- 所有 LLM 输出经过安全/合规层校验；
- 保留 LLM 推理和输出日志用于合规审计。

### 7.2 性能

- IM 消息发送延迟 < 5 秒（正常网络条件下）；
- Web Disease Card / Brief 页面首次加载 < 3 秒；
- PDF 生成 < 10 秒。

### 7.3 可用性

- 患者无需下载 App，全部交互在 WhatsApp + Web 完成；
- 每轮问答控制在 30 秒内完成；
- Web Disease Card / Brief 页面优先适配移动端。

### 7.4 可维护性

- RAG Corpus、Disease Profile、对话风格、Brief 模板可配置化；
- 记录 IM 平台 Webhook 日志；
- 记录 LLM 调用、Planner 决策、安全校验日志。

## 8. 成功指标

### 8.1 北极星指标

- **7 天记录完成率**：完成至少 4 次检查点的用户比例。目标 ≥ 60%。
- **Disease Card 查看率**：7 天后打开 Disease Card 的用户比例。目标 ≥ 70%。
- **Brief 生成率**：完成 7 天记录后成功生成 Brief 的比例。目标 ≥ 80%。
- **Brief 被带到复诊的比例**：用户自述带到复诊或分享给医生的比例。目标 ≥ 40%。

### 8.2 辅助指标

| 指标 | 目标 |
|------|------|
| WhatsApp 开启率 | ≥ 70% |
| 每次检查点完成时间 | ≤ 60 秒 |
| 用户 7 天后 NPS | ≥ 30 |
| 预订 4 周计划的比例 | ≥ 10% |
| 医生/护士认为 Brief 有价值 | ≥ 3/5 |
| 异常信号被正确识别并处理的比例 | ≥ 90% |

## 9. 合规与安全红线

1. **不诊断**：产品不能判断用户是否有哮喘发作、是否需要调整用药。
2. **不替代医生**：所有输出必须明确标注「患者自报信息，供复诊参考」。
3. **不做临床量表推断**：不能从稀疏非结构化回答中计算 ACT、GINA 等临床分数；仅当问题本身为量表时展示量表值。
4. **急救边界**：任何暗示急性发作的消息必须引导用户拨打急救电话或联系医生。
5. **数据最小化**：只收集生成 Disease Card 和 Brief 所必需的数据。
6. **透明同意**：首次使用前必须获得明确的隐私同意。
7. **医生端不通过 WhatsApp 接收临床信息**：MVP 通过 Web / PDF 传递；Email 分享为 MVP 后期扩展。
8. **可审计**：保留 LLM 决策、安全校验、数据访问日志。

## 10. 风险与假设

| 风险 | 影响 | 缓解措施 |
|------|------|----------|
| 用户不愿意在 WhatsApp 中回答健康问题 | 高 | 强调数据存放在安全 Web 页面，Disease Card 设计美观可信 |
| WhatsApp Business API 审核或模板限制 | 中 | 提前准备模板，准备备用 SMS/Email 通道 |
| 医生不认为 Brief 有价值 | 高 | 早期招募 5 位医生反馈，快速迭代模板 |
| 用户误以为 AI 在诊断 | 高 | 所有消息明确边界，Disease Card 和 Brief 都含免责声明 |
| LLM 输出不稳定或不安全 | 高 | 安全/合规层校验，人工审核 Corpus，保留日志 |
| 数据合规成本超出预期 | 中 | 使用最小化数据，咨询当地隐私顾问 |
