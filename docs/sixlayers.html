📊 一次消息请求的完整数据流

L1 读懂患者说了什么：提取意图/症状/异常标记
↳ L2 把原始消息 + 感知结果写入 Event / Observation 表（持久化）
↳ L3 从 RAG 语料库检索相关的哮喘医疗知识片段
↳ L4 根据患者状态 + RAG 知识决定下一个问题（ask / end_session）
↳ L5 把 planner 决策渲染成 WhatsApp 消息（文本 / 按钮）
↳ L6 检查诊断/治疗语言，追加急救免责声明，决定放行或改写

📅 用户旅程时间线

产品 MVP 的完整用户旅程，从首次接触到生成 Brief，贯穿六层引擎。

阶段一：Onboarding（Day 0）

用户 发送 START ASTHMA → L1 识别 intent=initiate → L2 创建 User + Cycle（ONBOARDING）
系统 发送欢迎语 + 隐私条款 → 用户 回复 AGREE → L2 更新 consentGiven=true
系统 收集昵称 / 年龄 / 复诊日期 / 用药基线 → L2 Cycle 转入 ACTIVE

阶段二：Check-in 周期（Day 1–28+）

调度器 BullMQ 到达 cycle.nextCheckinAt → L4 → L5 Planner 生成第 1 题 → Dialogue 渲染按钮消息
用户 点击按钮或自由文本回答 → L1 → L2 识别意图 + 提取 observation → 持久化 → L4 重新规划（预算 -1） → L5 → L6 渲染 + 安全校验 → 出站
↑ 重复最多 3 次（3 题预算，或进入异常模式追加）
end_session Budget 耗尽或全部 topic 覆盖 → L2 CheckIn status = COMPLETED → L2 生成 NarrativeSummary（session） → L2 生成 Disease Card → 调度器 安排下一次 check-in（48h/72h A/B）

用户主动记录（两次 Check-in 之间）

用户 主动发消息（非 check-in 中） → L1 → L2 提取 observation → 持久化 → L5 "已添加到记录"
ℹ️ 不创建 CheckIn 记录，不消耗预算，不生成 NarrativeSummary/Disease Card
ℹ️ 跨周期回复：系统询问 → 用户确认 YES → 保存到最近 cycle

阶段三：周期结束 → Brief 生成

系统 4 周计划到达 Day 28 → L2 Cycle status = COMPLETED → 用户 CONTINUE → 新 cycle / STOP → 结束
用户 复诊前打开 Brief 链接 → Web 查看 Brief HTML / 下载 PDF → 医生 60 秒内阅读结构化摘要

系统命令（任何时候）

HELP 显示所有可用命令 STOP 暂停提醒 + 取消周期 DELETE MY DATA GDPR 硬删除所有数据 EXPORT MY DATA 生成 7 天有效 JSON 导出
L1
感知层 Perception
解析患者输入，提取意图、症状和异常信号
▶

作用：把患者的 WhatsApp 消息（自由文本或按钮回复）解析成结构化数据。无论有没有 LLM（OpenAI），这层都必须运行。

🔧 技术细节

文件：packages/engine/src/perception.ts → perceive()

处理顺序（短路逻辑）：

  1. 接口初始化：填充 messageId、timestamp、traceId（UUID），默认 intent = answer
  2. 修正意图检测：6 种正则匹配 "I was wrong / actually / correction" 等 → intent: "correction"
  3. 系统指令（疾病无关）：完全由规则处理 — START <disease>（泛化，默认 asthma）、AGREE、SKIP / OK、STOP、DELETE MY DATA、EXPORT MY DATA、HELP、CONTINUE、YES / Y / ADD TO LAST RECORD / RECENT 等直接返回对应 intent
  4. 按钮映射：16 种哮喘按钮 ID → Observation 对象（症状 x4、用药 x4、活动 x2、依从性 x2、触发因素 x4）
  5. 上下文感知：若 context.checkInActive === false，将自由文本意图设为 question（用户主动咨询）；可配合 sessionObjective 注入 LLM prompt
  6. 安全启发式（始终运行，5 类检测）：
    • 严重症状语言：severe / can't breathe / chest tightness / blue lips / peak flow / no relief → high safety flag
    • 不良反应：rash / swelling / vomiting / dizziness / palpitations → medium anomaly
    • 复合信号冲突：3+ 维度同时恶化（worse + night + reliever + activity）→ medium anomaly
    • 严重度升级：sudden / rapidly / much worse / worst ever → high anomaly
    • 用药管理违规：reliever ≥ 3 次 + 未提及 controller → medium anomaly
  7. RAG 知识检索（L1 内嵌）：调用 loadDiseaseCorpus(disease) + searchCorpus(corpus, text, {topK:3})，为 LLM prompt 注入领域知识和为规则回退提供关键词
  8. LLM 路径（可选）：调用 perceiveWithLlm()，构建以下 System Prompt（含 RAG 上下文 + 会话状态）发送给 LLM，要求返回 JSON，temperature 0.2：
    You are the perception layer of CareMemory. Current disease context: {disease}.
    {session_context_block — 注入当前 check-in 状态和目标}

    Analyse the patient's free-text message and return ONLY valid JSON matching this schema:
    {
      "intent": { "primary": "answer" | "question" | "adverse_event" | "help" | "stop" | "delete_data" | "export_data", "confidence": number },
      "extractedObservations": [
        { "category": "symptom"|"medication"|"trigger"|"function"|"adverse_event"|"subjective"|"question",
          "concept": string, "value": any,
          "attributes": { "severity"?: string, "frequency"?: string, "duration"?: string } }
      ],
      "anomalies": [
        { "type": "management_rule_violation"|"compound_signal_conflict"|"possible_adverse_event"
          |"severity_escalation"|"pattern_contradiction",
          "description": string, "severity": "low"|"medium"|"high" }
      ],
      "safetyFlags": [
        { "type": string, "riskLevel": "none"|"low"|"medium"|"high", "description": string }
      ]
    }
    {rag_context_block — top-3 RAG 检索结果注入}

    Guidelines:
    - Use "adverse_event" as primary intent when the patient reports a suspected drug reaction or side effect.
    - Use "question" when the patient is asking their own question, not answering a check-in prompt.
    - For each extracted observation, include "attributes" with severity, frequency, or duration when implied.
    - Detect anomalies by type: management_rule_violation, compound_signal_conflict, severity_escalation, pattern_contradiction.
    Do not diagnose or give treatment advice. Use concept names like "nighttime_symptoms", "reliever_use",
    "activity_limitation", "trigger_exposure", "controller_adherence" when applicable.

    User Message: 原文直接作为 user role 消息发送。LLM 失败或未配置时降级到规则路径（关键词提取 + free_text_response）。

  9. 规则兜底（增强）：关键词提取（cough / wheeze / chest tightness / breathlessness / nighttime / reliever / controller）+ RAG 医学术语 + 数值提取；无匹配时回退为 subjective / free_text_response

🔀 安全路由（safetyFlags → 下游分岔）

L6 快速安全路径（engine.ts L499-537）：

若 perception.safetyFlags 中任意一条 riskLevel === "high" → 直接返回安全提示，跳过 L4 Planner：

"I'm sorry you're struggling. If you're having severe breathing problems, call 999 or follow your asthma action plan now..."

LLM 调用已记录（llm_call event ✅）+ 安全消息已落库（outbound_message event ✅）— 但 Planner 不会被调用，患者看不到 LLM 规划的追问。

异常模式路由（riskLevel = medium）：

若所有 safetyFlag 风险 ≤ medium 且存在 anomaly → Planner 进入 exception mode，追问最多 3 个澄清问题。患者看到的是 Planner 产生的自然语言追问，而非固定安全提示。

实际案例对比：
输入LLM 判定路由患者看到
"昨晚咳嗽严重，服用了急救药物"severe_symptom: highL6 快速路径呼叫 999 安全提示
"前天晚上也咳嗽了，服用急救药物"symptom_medication_combination: mediumPlanner 异常模式"Can you tell me more about what happened?"

L2
记忆层 Memory
所有数据的持久化读写中枢；GDPR 合规接口
▶

作用：把所有事件（入站消息、感知结果、规划结果、LLM 调用、出站消息）持久化到 PostgreSQL。L2 是引擎唯一直接操作数据库的层。

🔧 技术细节

文件：packages/engine/src/memory.ts

核心函数

函数用途关键细节
saveInboundMessage()保存患者原始消息按 platformMessageId 去重，保证幂等性
saveObservations()保存 L1 提取的结构化 observation每条 ob 绑定到 cycle + event；支持 superseded=true 修正
saveOutboundMessages()保存引擎发出的回复生成 SHA256 的 idempotencyKey，状态为 pending，dispatch 层再更新为 sent/failed
savePerceptionEvent()存档完整 L1 结果Event type: observation_extracted
savePlannerEvent()存档完整 L4 结果Event type: state_updated
saveLlmCallEvent()LLM 调用审计日志记录 model / input / output / token usage，GDPR 导出时默认排除
deleteUserData()GDPR 硬删除按依赖顺序删除（brief → diseaseCard → observation → event → checkIn → cycle → user）
exportUserData()GDPR 数据导出格式 carememory-gdpr-export-v1，默认排除 llm_call 事件

幂等性策略

idempotencyKey = "out:" + userId + ":" + timestamp + ":" + salt + ":" + sha256(text).slice(0,16)

salt 来自 inboundEventId 或 checkIn.id，确保不同业务上下文产生的 outbound 不会冲突。

被谁调用：引擎主流程 engine.ts 在所有阶段穿插调用 L2 函数

L3
知识检索层 RAG
基于 TF-IDF 关键词检索的哮喘医学知识库
▶

作用：在 Planner 做决策前，从预编译的哮喘医学语料库中检索相关知识片段，注入到 Planner 的 reasoning 中。L3 不独立接收消息，而是 L4 在需要时拉取。

🔧 技术细节

文件：packages/rag/src/index.ts

语料库

packages/rag/documents/asthma/
├── medical-overview.md      ← 哮喘的病理生理、分型、评估量表
├── care-strategy.md         ← UK 哮喘管理指南、阶梯治疗方案
├── conversation-patterns.md ← 对话模式（如何询问症状、如何回应异常）
└── safety-rules.md          ← 安全红线规则

这些 Markdown 文件通过 pnpm corpus:reindex 预编译为 packages/rag/src/corpus.ts（TS 模块），运行时直接加载，无需外部向量数据库。

检索算法

searchCorpus(corpus, query, { topK: 2 })
// 1. Tokenize query → 去停用词（≤2 chars）
// 2. 对每个 corpus section 计算 TF-IDF 得分
// 3. 标题命中额外 +2 分
// 4. 返回 topK 个最相关 section

检索触发时机：Planner 每次执行规则逻辑时都会调用 searchCorpus()，查询词为 "asthma {intent} {topic}"。

与 L4 的关系

L3 是 L4 的附属模块——Planner 构造 reasoning 时会拼接 L3 返回的知识摘要：

reasoning: "Cycle day 3. Retrieved: medical-overview: Asthma control assessment uses 3 core dimensions - nighttime symptoms, reliever use, activity limitation. Asking nighttime_symptoms to fill control assessment."
L4
规划层 Planner
根据患者状态和对话上下文，决定下一个问题和行动计划
▶

作用：基于患者当前状态（cycle day / 已回答哪些问题 / 是否有异常）+ L3 知识片段 + 预算约束，决定系统下一步行为。

🔧 技术细节

文件：packages/engine/src/planner.ts → plan()

决策优先级（从上到下短路判断）

  1. 异常模式（inExceptionMode）：检测到中高风险异常时触发，最多追加 3 个澄清问题（exception_clarification → exception_impact → exception_action），然后 end_session + 安全提示
  2. 不良事件：recentObservations 中有 adverse_event → 立刻触发 type: "safety_response"（建议联系 GP/111）
  3. 预算耗尽：budgetRemaining ≤ 0 → end_session
  4. 主题覆盖：从 CHECKIN_QUESTIONS 中按序选取未问过的 topic → type: "ask"
  5. 全部覆盖：所有 3 个 topic 问完 → end_session

三个核心检查问题（CHECKIN_QUESTIONS）

Topic问题目的选项预算消耗
nighttime_symptoms夜间咳嗽/喘息None / Mild / Disturbed sleep / Woke me up1
reliever_use缓解药物使用次数0 / 1 / 2 / 3+1
activity_limitation日常活动受限No / Yes1

LLM 路径

systemPrompt 要求返回 JSON（reasoning + nextAction + safetyFlag）
输入：disease, cycleDay, budgetRemaining, recentObservations, askedTopics, currentIntent
temperature: 0.2
安全保护：LLM 选择已问过的 topic 会被校验拦截并抛出异常 → 降级到规则逻辑

📤 输出：PlannerOutput

{
  reasoning: "Cycle day 3. Retrieved: ...",
  sessionObjective: "Track asthma control dimensions over the last 48 hours.",
  nextAction: {
    type: "ask" | "end_session" | "safety_response" | "inform",
    topic, purpose, expectedResponseType, options, budgetCost
  },
  safetyFlag: "none" | "low" | "medium" | "high",
  updatePatientState: {}
}

传递给谁：L5（渲染消息）+ L6（安全检查）

L6
安全层 Safety
拦截诊断/治疗语言，追加合规免责声明，分级风险管理
▶

作用：对所有即将发给患者的消息做最终安全检查。L6 是最后一道防线，确保任何消息都不会包含诊断性语言或缺少急救提示。

🔧 技术细节

文件：packages/engine/src/safety.ts → safetyCheck()

拦截规则（PROHIBITED_PATTERNS）

"you (have|are having) (an asthma attack|a severe attack|an attack)"
"you should (increase|decrease|stop|start|change) (your medication|your inhaler|the dose)"
"take \\d+ puffs? (of|from) your"
"you need (steroids|antibiotics|a nebulizer)"
"your asthma is (severe|uncontrolled|life-threatening)"

命中任一模式 → approved: false, riskLevel: high，消息被改写为安全兜底文本。

免责声明追加

触发条件追加内容
消息包含 asthma / wheez / inhaler / breath / reliever / controller"If you're having severe breathing problems, call 999 or follow your asthma action plan."
消息包含 health / symptom / medication / doctor / clinic / visit"This is based on patient-reported information only and is not medical advice."

引擎级聚合

engine.ts 中的 safetyWrapWithSummary() 对多条 outbound 消息逐一调用 safetyCheck()，然后：

  • 取所有结果中最高的 riskLevel
  • 合并去重所有 requiredAddendums
  • 任一消息未通过 → 整体 approved: false
  • 结果写入 EngineTrace.safety

📤 输出：SafetyResult

{
  approved: boolean,
  riskLevel: "none" | "low" | "medium" | "high",
  requiredAddendums: string[],
  blockReason?: string
}

最终去向：安全校验通过的消息 → L2 持久化 → dispatch 发送；未通过的消息被替换为安全兜底文本

⚙️ 关键架构设计

崩溃恢复入站消息按 platformMessageId 去重；出站消息带 idempotencyKey
LLM 配额用户级每日软上限（Redis 原子计数 + 当天过期）；超出后自动降级规则逻辑 + 3 分钟内不重试
A/B 实验基于 userId hash 的稳定分桶，控制 check-in 频率（48h/72h）和对话风格（v1/v2）
异常模式中高风险异常 → 进入 exception mode → 最多 3 个澄清问题 → 结束 + 安全提示 → 写入 Disease Card Adverse Events
迟到回答同一 cycle 内无 active check-in → 接受为补充更新；跨 cycle → 系统确认后保存到最近 cycle
用户修正L1 识别修正意图 → 原 observation 标记 superseded=true → 新 observation 替代
4 周周期延续28 天到期 → 提示 CONTINUE → 用户回复后创建新 cycle
GDPR硬删除（DELETE MY DATA）+ 7 天有效导出链接（EXPORT MY DATA，默认排除 LLM 审计）
