🔍 六层详细说明

L1
感知层 Perception
意图识别、Observation 提取、安全标记、异常检测、系统命令硬编码
▶

📥 输入 / 📤 输出

输入用户原始消息 (文本 / WhatsApp 按钮) + 当前会话上下文 (是否有活跃 check-in、session 目标)
输出 → 引擎意图 (intent)、观测项列表 (observations)、安全标记 (safetyFlags)、异常信号 (anomalies)
输出 → L2整份 PerceptionResult (含 traceId)，由引擎调用 L2 写入 Event + Observation 表

🧠 判断逻辑（按优先级依次匹配）

1
修正意图检测
用户说 "I was wrong" / "actually" / "correct that" → 判定为修正意图 (correction)，后续 L2 会将旧 observation 标记为 superseded
2
系统命令匹配（8 条硬编码，不走 LLM）
用户输入判定的意图是否生成 observation引擎如何响应
START ASTHMAinitiate是 (system_intent)创建新用户 / 新 Cycle
AGREEconsent是 (system_intent)更新用户同意状态，推进 onboarding
STOPstop否暂停 Cycle，停止提醒
HELPhelp是 (system_intent)返回命令列表 + 急救提示
DELETE MY DATAdelete_data否硬删除用户全部数据
EXPORT MY DATAexport_data否生成 7 天有效导出链接
SKIPskip否跳过当前问题
CONTINUEcontinue_cycle否结束旧周期，开始新 PLAN_4_WEEK
3
按钮回复解析
WhatsApp 交互式按钮 (如用药选择) → 直接映射为对应 concept 的 observation (如 medication/reliever_use)
4
LLM 理解（主路径）
将用户消息发给 LLM → 结构化输出：意图类型 + 提取到的症状/用药/触发因素等 observation + 安全风险等级 + 是否检测到异常
5
规则降级（无 LLM 时）
用 RAG 语料库中的医学关键词 + 正则表达式从消息中提取 observation，意图默认为 "answer"

📊 提取的 Observation 分类

分类说明示例 (用户说了什么 → concept)
symptom症状报告"I've been wheezing" → wheezing
medication用药记录"used my blue inhaler twice" → reliever_use
trigger触发因素"after I went running" → exercise
function功能影响"couldn't sleep well" → sleep_disturbance
adverse_event不良事件异常模式自动写入 → exception_concern
subjective主观描述迟到回答跨周期保存 → free_text_response
system_intent系统命令"START ASTHMA" → start_asthma

⚠️ 安全标记 (SafetyFlags) — 引擎据此做快速路径判断

风险等级引擎行为
high立即返回紧急响应 ("call 999...")，不进入 L4 Planner，整条链路短路
medium若同时存在活跃 check-in → 触发异常模式，进入最多 3 轮澄清追问
low / none正常进入 L4 规划流程

🚨 异常检测 (Anomalies) — 引擎据此进入异常模式

严重程度引擎行为
medium / high有活跃 check-in 时 → 进入异常模式：check-in 标记 inExceptionMode=true，最多 3 个澄清问题，结束后写入 DiseaseCard Adverse Events 模块
low / none正常流程
L2
记忆层 Memory
事件/observation 持久化、修正处理、NarrativeSummary 生成、GDPR 操作、幂等性
▶

📥 输入 / 📤 输出

写输入来自引擎主流程：用户原始消息 + L1 的 PerceptionResult + L5/L6 的出站消息
读输出向 L4 提供近期 observations (getRecentObservations)；向自己 (L2) 提供 observations 用于 DiseaseCard 聚合
写 DB5 张核心表：Event、Observation、NarrativeSummary、DiseaseCard、Brief

📋 核心数据表 — 何时生成、依赖谁的输出

表何时写入写入内容来源后续被谁消费
Event (inbound_message)每次收到用户消息用户原始消息 (platformMessageId 去重)审计追溯、GDPR 导出
Event (observation_extracted)L1 感知完成后L1 的 PerceptionResult (不含 observations 详情)审计追溯
Event (outbound_message)每次准备发送回复L6 校验后的 OutboundMessage (idempotencyKey 防重复)dispatch 层取出发送
Event (llm_call)每次 LLM 调用后L1 或 L4 的 LLM 请求/响应 (model + tokens)审计 + 配额统计
Observation每次用户输入L1 的 extractedObservations (结构化健康数据)L4 (近期上下文) + DiseaseCard 聚合
NarrativeSummary每次 check-in 结束 (end_session)本周期 L2 的 observations (LLM 总结为自然语言)L4 (下次 check-in 上下文)
DiseaseCard每次 check-in 结束 (end_session)本周期 L2 的 observations (过滤 superseded=false, version 递增)Web 页面展示 + Doctor Brief 导出

🔄 用户修正处理

步骤谁触发做什么
1. 检测L1识别用户修正意图 ("I was wrong, actually...")
2. 写入L2创建新的 observation 记录 (替代旧值)
3. 标记L2查询同 concept 的旧 observation → 标记 superseded=true
4. 过滤L2 + L4后续所有查询自动加 WHERE superseded=false，旧值不可见

🧹 数据生命周期

用户删除发送 DELETE MY DATA → L1 识别 → 引擎调用 L2 按依赖顺序硬删除 (brief → diseaseCard → narrativeSummary → observation → event → checkIn → cycle → user)，不可恢复
用户导出发送 EXPORT MY DATA → L1 识别 → 引擎调用 L2 生成 7 天有效下载链接，格式 carememory-gdpr-export-v1，默认排除 LLM 调用审计日志
L3
知识层 RAG
哮喘疾病知识语料库，关键词检索增强 Planner 推理
▶

文件：packages/rag/src/index.ts + corpus.ts (预编译)

Corpus 按疾病分段，每段含 title + content + keywords。检索时做关键词匹配，返回相关 CorpusSection[] 注入 Planner 的 ragContext。重新索引：pnpm corpus:reindex

L4
规划层 Planner
LLM 驱动动态问题规划，含异常模式、预算管理、RAG 增强
▶

文件：packages/engine/src/planner.ts → plan()

输入：PlannerInput { patientContext (disease, cycleDay, narrativeSummary, recentObservations, openIssues), conversationContext (intent, questionsAsked, budgetRemaining, inExceptionMode, exceptionQuestionsAsked), temporalContext, ragContext }

输出：PlannerOutput { reasoning, sessionObjective, nextAction: { type(ask/inform/end_session), topic, purpose, budgetCost }, safetyFlag, updatePatientState }

异常模式下最多 3 个澄清问题，结束后安全提示。预算管理控制每轮 check-in 问题数量。LLM 模式使用 LLM_REASON_MODEL 配置的推理模型。

L5
对话层 Dialogue
结构化 plan → 平台无关消息 (文本 / 按钮)
▶

文件：packages/engine/src/dialogue.ts → renderMessage()

输入 PlannerOutput，输出 OutboundMessage { userId, conversationContext, content: { type: "text"|"buttons", text, buttons? } }。WhatsApp Adapter 负责将 OutboundMessage 转为实际发送格式。

L6
安全层 Safety
拦截诊断/治疗语言，追加合规免责声明，分级风险管理
▶

拦截规则

"you (have|are having) (an asthma attack|a severe attack|an attack)"
"you should (increase|decrease|stop|start|change) (your medication|your inhaler|the dose)"
"take \\d+ puffs? (of|from) your"
"you need (steroids|antibiotics|a nebulizer)"
"your asthma is (severe|uncontrolled|life-threatening)"

免责声明追加

触发条件追加内容
含 asthma / wheez / inhaler / breath / reliever / controller"If you're having severe breathing problems, call 999..."
含 health / symptom / medication / doctor / clinic / visit"This is based on patient-reported information only..."

⚙️ 关键架构设计

崩溃恢复入站消息按 platformMessageId 去重；出站消息带 idempotencyKey
LLM 配额用户级每日软上限（Redis 原子计数 + 当天过期）；超出后自动降级规则逻辑
A/B 实验基于 userId hash 的稳定分桶，控制 check-in 频率（48h/72h）和对话风格
异常模式中高风险异常 → exception mode → 最多 3 个澄清问题 → 安全提示 → 写入 Adverse Events
迟到回答同一 cycle 内无 active check-in → 接受为补充更新；跨 cycle → 用户确认后保存
用户修正L1 识别修正意图 → 原 observation 标记 superseded=true → 新 observation 替代
4 周周期延续28 天到期 → 提示 CONTINUE → 用户回复后创建新 cycle
GDPR硬删除 + 7 天有效导出链接（默认排除 LLM 审计）
