## 总览

L2 患者记忆层的定义（综合 func-spec + tech-spec）：

> **职责**：维护不可变事件日志、观察记录、多级 narrative summary、Disease Card。
> **请求流中的位置**（tech-spec §8.1）：L1 → **L2 写入 Event + Observation；更新 Narrative Summary** → L6 → L3 → L4 → L5 → L6

L2 的产出向下游（L4 Planner）提供 `patientContext`（observations、narrativeSummary、openIssues），向上游（用户/医生）提供 Disease Card 和 Brief。

| 子组件 | func-spec 引用 | 完成度 | 关键 gap |
|--------|---------------|--------|----------|
| Event Log | §5.1 | ✅ 完整 | — |
| Observations | §5.1 + §5.2 | ✅ 完整 | 字段命名差异（eventId vs sourceEventId） |
| Narrative Summaries | §5.1 + §5.3 | ⚠️ 部分 | 缺 cycle + longitudinal scope；keyObservationIds 是占位符 |
| Disease Card | §5.1 + §9 | ⚠️ 部分 | 缺 4 个模块；无增量更新；无 LLM 生成；version=1 |
| Brief | §10 | ⚠️ 部分 | 无 LLM 生成；无事件时间线 |
| Planner 结果存储 | §6.2 + tech-spec §8.1 | ❌ 缺失 | updatePatientState 不被处理；openIssues 无存储 |
| 数据保留策略 | tech-spec §5.2 | ❌ 缺失 | 90 天清理、LLM 日志保留、删除审计 均未实现 |

## 一、Event Log — ✅ 已完成

**func-spec 要求**（§5.1）：不可变事件日志，类型包括 `inbound_message` / `outbound_message` / `observation_extracted` / `state_updated` / `llm_call` / `safety_check`。

**tech-spec 要求**（§5.1 + §11.3）：支持 `platformMessageId` 去重、`idempotencyKey` 幂等性、审计日志。

**代码验证**：
- `memory.ts` 中 `saveInboundMessage`、`saveOutboundMessages`、`savePerceptionEvent`（observation_extracted）、`savePlannerEvent`（state_updated）、`saveLlmCallEvent` 全部存在
- Prisma `Event` 模型的 `EventType` enum 覆盖所有指定类型，额外扩展了 `checkin_scheduled` / `checkin_sent` / `checkin_completed` / `user_action`
- `platformMessageId` 索引存在，webhook 去重使用
- `idempotencyKey` 唯一约束存在，dispatch 层检查 `pending`/`sent`/`failed` 状态
- `traceId` 字段存在但**未在 engine 中赋值**（seed 随机值？待确认）

**结论**：Event Log 满足 func-spec 和 tech-spec，无 gap。

## 二、Observations — ✅ 已完成

**func-spec 要求**（§5.1 + §5.2）：

```typescript
interface Observation {
  observationId: string;
  userId: string;
  cycleId: string;
  timestamp: Date;
  sourceEventId: string;       // func-spec 命名
  category: "symptom" | "medication" | "trigger" | "function" | "adverse_event" | "subjective" | "question" | ...;
```

**代码验证**：
- `saveObservations` 使用 `eventId` 而非 `sourceEventId` — 功能等价，命名差异
- Prisma `Observation` 模型支持所有字段 + `superseded` / `supersededById`（D18 用户修正）
- `getRecentObservations` 默认过滤 `superseded: false`
- `supersedePreviousObservations` 已实现并接入 engine（第 545–548 行）
- ObservationCategory enum 覆盖 func-spec 全部类别

**结论**：基本完成。唯一差异是 `eventId` vs `sourceEventId` 命名。

## 三、Narrative Summaries — ⚠️ 部分完成

### ✅ 已完成

- **Session-level narrative**：`generateSessionNarrativeSummary()` 在每次 check-in end_session 时生成
- **Prisma schema**：`NarrativeSummary` 模型包含 scope、content、keyObservationIds、model 等所有字段
- **系统提示词**：包含"不诊断、不提供治疗建议"的安全约束
- **tech-spec §6.1 模型匹配**：使用 perception 层模型（GPT-4o-mini），符合推荐

### ❌ 未完成

| # | Gap | 描述 | 依据 | 严重度 |
|---|------|------|------|--------|
| G1.1 | **缺 cycle 级摘要** | `scope: "cycle"` 从未被使用。没有函数生成跨 check-in 的周期级叙事摘要 | func-spec §5.1: `scope: "session" \| "cycle" \| "longitudinal"` | **高** |
| G1.2 | **缺 longitudinal 级摘要** | `scope: "longitudinal"` 从未被使用。没有函数生成跨 cycle 的纵向叙事摘要 | 同上 | **中** |
| G1.3 | **keyObservationIds 是假占位符** | 代码第 197 行 `observations.map((_, i) => \`${i}\`)` — 传入的是数组索引，不是数据库真实 observation ID | func-spec §5.1: `keyObservations: string[]` — 明确要求"引用的 observation IDs" | **中** |
| G1.4 | **"异步触发"是假的** | engine.ts 第 657 行直接 `await` 调用，不是真的异步/队列任务 | func-spec §5.3 step 3: "异步触发 Narrative Summary 更新"；tech-spec §8.2: "使用 BullMQ，支持重试和死信队列" | **低** |
| G1.5 | **Planner 的 updateNarrative flag 被忽略** | `safetyResponse()` 和 `endSession()` 都设 `updateNarrative: true`，但 engine 从不检查这个 flag（不过 engine 在 end_session 时会独立触发 narrative 生成，因此影响不大） | func-spec §6.2: `PlannerOutput.updatePatientState.updateNarrative?: boolean` | **低** |

### 为什么 G1.1 是 gap

Planner 的 `narrativeSummary` 输入永远只有**最后一次 check-in 的 session 摘要**（engine.ts 第 590–593 行 `findFirst orderBy: generatedAt desc`）。func-spec D5 明确说 Narrative Summaries 分三级，设计意图是 Planner 可以根据需要加载不同粒度的患者状态。缺 cycle 级意味着 Planner 看不到整个周期的汇总趋势。

## 四、Planner 结果存储 — ❌ 缺失

### ✅ 已做的

Planner 的完整输出作为 JSON blob 通过 `savePlannerEvent()` 写入 Event Log（类型 `state_updated`），保证审计痕迹。

### ❌ 未做的（Gap #2）

| # | Gap | 描述 | 依据 | 严重度 |
|---|------|------|------|--------|
| G2.1 | **newObservations 未写入 Observation 表** | `PlannerOutput.updatePatientState.newObservations` 从未被 engine 提取并调用 `saveObservations`。Planner 即使推理出了新的 observation，engine 也忽略 | func-spec §6.2: `updatePatientState.newObservations?: Observation[]` | **中** |
| G2.2 | **addOpenIssue / resolveOpenIssue 无存储** | `updatePatientState.addOpenIssue / resolveOpenIssue` 从未被处理。且**没有数据库表或字段用于存储 openIssues** | func-spec §6.2: `addOpenIssue?: string; resolveOpenIssue?: string` | **高** |
| G2.3 | **openIssues 永远是空数组** | `PlannerInput.patientContext.openIssues` 两次调用处（engine.ts 第 601、766 行）都硬编码为 `[]`，不从数据库加载 | func-spec §6.1: `openIssues: string[]` | **高** |
| G2.4 | **Planner LLM 系统提示词禁止了 updatePatientState** | planner.ts 第 160 行 `"updatePatientState": {}` — system prompt 的 JSON schema 中写死了空对象，LLM 不会生成任何 `updatePatientState` 内容 | 与 func-spec §6.2 冲突 | **中** |

### 为什么是 gap

G2.2 + G2.3 意味着 func-spec 精心设计的"open issues 跟踪"机制完全不存在：

- Planner 不知道当前有哪些 open issues（因为总是 `[]`）
- Planner 无法添加新 issue（因为没有存储）
- Planner 无法关闭 issue（因为没有存储）
- 整个 `openIssues → addOpenIssue/resolveOpenIssue` 闭环从接口到实现全部是空中楼阁

最简单的修复方案：给 `Cycle` 加一个 `openIssues: String[]` 字段（JSON 存储）。

|---|------|------|------|--------|
| G3.1 | **缺 Adherence（依从性）模块** | Disease Card 没有展示 controller inhaler 等维持用药的依从性。这是医生复诊前最需要的信息之一 | func-spec §9.2 模块表：Adherence — MVP 7 天版需"简单计数" | **高** |
| G3.2 | **缺 Symptom-Treatment Relationship 模块** | 没有分析症状与用药之间的关联 | func-spec §9.2 模块表 | **低**（MVP 7 天只需"初步观察"） |
| G3.3 | **缺 Subjective Changes 模块** | 患者主观感受变化（"我觉得好一点了"等）没有被整理为独立模块 | func-spec §9.2 模块表：Subjective Changes — 文本摘要 | **中** |
| G3.4 | **7 天版 vs 4 周+ 版无区分** | func-spec 为每个模块设计了双列粒度对比，当前永远输出同一套内容 | func-spec §9.2 双列对比表；PRD §3.2 记录周期 | **中** |
| G3.5 | **模块置信度缺失** | 只有 `symptom_trend` 有条件置信度（0.6/0.9），其他模块无 `confidence` | func-spec §9.3: "数据稀疏时显示置信度提示" | **中** |
| G3.6 | **只 end_session 生成，不做增量更新** | Disease Card 只在 check-in 完成时（end_session）创建一次。用户做 3 次后退出则无 card | func-spec §9.4: "每次 check-in 后增量更新 headline 和近期模块"；验收标准 §17.3: "每次 check-in 后 Disease Card 可更新" | **高** |
| G3.7 | **version 硬编码为 1** | 没有版本递增机制 | func-spec §5.3 step 5: "所有更新保留版本历史" | **低** |
| G3.8 | **没有 LLM 生成** | tech-spec §6.1 推荐 Disease Card 使用 GPT-4o / Claude 3.5 Sonnet。当前实现是纯规则的，模块内容来自静态组合 | tech-spec §6.1 LLM 使用场景表；func-spec §10.3: "由 LLM 生成" | **中** |
| G3.9 | **Web 页缺失数据来源和置信度提示** | func-spec §15.1 要求"显示数据来源和置信度提示"，Web 端 DiseaseCardModule 组件未实现此功能 | func-spec §15.1；相同要求见 tech-spec §9.1 | **中** |
| G3.10 | **control_status 的 confidence 未传递到 Web** | `assessControl()` 返回 `{ status, reason, confidence }`，但 `generateDiseaseCard()` 没有把 `confidence` 放进模块的顶层 `confidence` 字段（模块的 confidence 是 undefined，只有 symptom_trend 有） | func-spec §9.3 | **低** |

### 为什么 G3.1 是最严重的

func-spec §9.2 把 **Adherence** 列为四大核心模块之一（与 Headline、Control/Status、Symptom-Trend 并列）。func-spec §10.2 的 Brief 结构中也把"用药/依从性"列为关键信号。当前仅收集 reliever 使用频率，没有结合 onboarding 用药基线判断依从性。

这意味着：**如果医生打开 Brief，看不到"患者有没有按时用 controller inhaler"**——而这是复诊时最有价值的患者自报信息之一。

### 为什么 G3.6 是结构性风险

如果用户只完成了 3 次 check-in，Disease Card 不会生成（只在 end_session 时 create）。但 func-spec §9.4 明确说"每次 check-in 后增量更新"——应该在每次 check-in 结束时 upsert Disease Card，而不是只 create 一次。
