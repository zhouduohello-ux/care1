Prisma schema 定义了 10 种 EventType，代码中实际使用的只有 5 种：

EventType状态应产生时机

outbound_message✅ 已使用—

位置：schema.prisma:105-115 定义了枚举；engine.ts 各操作用户/check-in 的地方未写 event；scheduler.ts 未写 checkin_scheduled/sent 事件

影响：Event Log 审计链路不完整，GDPR 导出缺少关键操作用户行为记录。

HIGH GAP-H2 HELP 命令不生成 observation，与其他系统命令行为不一致 Observation 表 ✅ FIXED

所有系统命令（START ASTHMA / AGREE / SKIP / STOP / DELETE / EXPORT / CONTINUE / YES）都会生成 system_intent 类 observation，但 HELP 仅设 intent 就返回。

位置：perception.ts:67-70 — 只设 result.intent = { primary: "help" }，没有 push observation

// perception.ts:67-70 if (upperText === "HELP") { result.intent = { primary: "help", confidence: 1.0 }; return result; // ← 不生成 observation } // 对比 STOP (perception.ts:57-65): result.extractedObservations.push(makeObservation("system_intent", "stop", {}));

影响：func-spec 要求每次用户输入都生成 observation，HELP 打破了这个约定。Disease Card 和 NarrativeSummary 缺少用户求助事件。

HIGH GAP-H3 DiseaseCard 生成时未过滤 superseded=true 的 observation Observation → DiseaseCard 链路 ✅ FIXED

DiseaseCard 生成查询获取所有 observation，包含已被用户修正废止的记录。与 decisions D18 和 sixlayers.html 中「下游只读取 superseded=false」的设计矛盾。

位置：engine.ts:705-706 — prisma.observation.findMany({ where: { cycleId: cycle.id } }) 缺少 superseded: false

// engine.ts:705-706 (当前) const allObservations = await prisma.observation.findMany({ where: { cycleId: cycle.id }, // ← 缺少 superseded: false }); // 对比 getRecentObservations (memory.ts:291-294) 正确过滤: where: { userId, cycleId, superseded: false }

影响：用户修正后的旧值仍出现在 DiseaseCard 中，患者看到的卡片内容不可信。

HIGH GAP-H4 platformMessageId 缺少数据库唯一约束 — 入站去重不可靠 Event 幂等性 ✅ FIXED

schema 中 platformMessageId 只有 @@index，没有 @unique。并发 webhook 投递可绕过应用层检查产生重复数据。

位置：schema.prisma:129 — platformMessageId String?；schema.prisma:140 — @@index([platformMessageId])，应为 @unique

影响：竞态条件可能导致同一条 WhatsApp 消息被处理两次，产生重复 observation 和 event。

🟡 中优先级 GAP（应修复）

MED GAP-M1 tokenUsage 从未填充（LLM API 返回的 usage 被丢弃） Event 审计 ✅ FIXED

LLM 客户端在 llm.ts:163-168 解析 API 响应时只提取 message.content，丢弃了 data.usage（prompt_tokens / completion_tokens / total_tokens）。审计回调收到 tokenUsage: undefined。

位置：llm.ts:163-168；perception.ts:222、planner.ts:182 调用 onLlmCall(model, input, output) 不传 tokenUsage

影响：LLM 成本可观测性缺失，admin metrics 无法统计实际 token 消耗。

MED GAP-M2 llmModel 字段存储层名而非实际模型名 Event 审计 ✅ FIXED

审计回调传递的 model 参数是 "perception" / "planner"（层标识符），而非实际模型名如 "gpt-4o-mini"。无法从 event 记录中知道实际用了哪个模型。

位置：perception.ts:222；planner.ts:182；engine.ts:51-78 resolveLlmClient 在内部解析模型但不暴露

MED GAP-M3 所有 event 缺少 traceId — 同一请求的事件无法关联 Event 调用链 ✅ FIXED

traceId 在 schema 中存在且 perception.ts:26 生成 UUID，但从未持久化到数据库。一次请求产生的所有 event（inbound → observation_extracted → llm_call → outbound）无法按请求关联。

位置：memory.ts 全部 5 个创建函数（saveInboundMessage / saveOutboundMessages / savePerceptionEvent / savePlannerEvent / saveLlmCallEvent）均未传 traceId

MED GAP-M4 高风险安全路径中 observation_extracted 事件成为孤儿 Observation → Event 链路 ✅ FIXED

当 L1 检测到 high-risk safety flag 后，引擎直接返回安全提示（engine.ts:534-549），跳过了 Planner 和 saveObservations。L1 已提取的 observation（如 adverse_event）只存在 event 表中，未写入 observation 表。

位置：engine.ts:534-549 — 高风险快速路径 return 前缺少 saveObservations

MED GAP-M5 迟到回答路径中 observation_extracted 事件成为孤儿 Observation → Event 链路 ✅ FIXED

与 GAP-M4 类似，迟到回答逻辑（engine.ts:555-574）在返回确认消息前也未调用 saveObservations。

位置：engine.ts:555-574 — 迟到回答 return 前缺少 saveObservations

MED GAP-M6 NarrativeSummary cycle 和 longitudinal 级别从未生成 NarrativeSummary 表 ⏳ 新功能

只实现了 scope: "session"。func-spec §5.1 定义的三级 scope（session / cycle / longitudinal）中 cycle 和 longitudinal 没有代码路径。

位置：memory.ts:189-198 — generateSessionNarrativeSummary 硬编码 scope，无 cycle/longitudinal 生成函数

MED GAP-M7 DiseaseCard 缺少「Symptom-Treatment Relationship」模块 DiseaseCard 表 ⏳ 新功能

func-spec §9.2 定义 9 个模块，generator.ts 只实现 8 个。第 4 个模块（症状-治疗关联分析，如 "nighttime symptoms + high reliever use → possible under-treatment"）无对应代码。

位置：disease-card/src/generator.ts — 无 symptom_treatment 生成逻辑；types.ts:4 — 缺少类型定义

MED GAP-M8 Brief 不会在周期结束时自动生成 Brief 表 ✅ FIXED

func-spec §10.1 列出 4 种触发条件（7 天结束 / 4 周结束 / 复诊前 / 用户请求）。当前只有「用户请求」通过 API 端点可用。周期结束时（engine.ts:687-744）生成 DiseaseCard 但不会生成 Brief。

位置：engine.ts:687-744 — end_session 流程只生成 DiseaseCard，不生成 Brief

MED GAP-M9 TRIAL_7_DAY cycle 永远不会转到 COMPLETED Cycle 状态机 ✅ FIXED

周期结束逻辑只检查 cycle.type === "PLAN_4_WEEK"（engine.ts:727-728）。TRIAL_7_DAY 类型的 cycle 会永远继续调度 check-in，不会自动结束。

位置：engine.ts:727-728 — if (cycle.type === "PLAN_4_WEEK" && cycleDay >= 28)，没有 7 天完成逻辑

MED GAP-M10 CheckIn 从未进入 SCHEDULED 或 MISSED 状态 CheckIn 状态机 ✅ FIXED

Check-in 直接以 SENT 状态创建（engine.ts:785-793），从不经过 SCHEDULED。24h 无回复后发送提醒（scheduler.ts:82-84）但不转 MISSED。check-in 状态永远只能是 SENT → COMPLETED 或 SENT → EXCEPTION。

位置：engine.ts:785-793 跳过 SCHEDULED；scheduler.ts:56-87 不转 MISSED
